This document lists the maintainers of the Forge project.

## Current Maintainers

- **Forge Team** (@forge-team)
  - **Role**: Project Lead & Architecture
  - **Responsibilities**: Overall project direction, architecture decisions, release management
  - **Contact**: forge-maintainers@example.com

## Maintainer Responsibilities

Maintainers are responsible for:

- **Code Review**: Reviewing and merging pull requests
- **Issue Triage**: Managing and responding to issues
- **Release Management**: Creating releases and managing versioning
- **Community Management**: Fostering a healthy community environment
- **Security**: Responding to security vulnerabilities
- **Documentation**: Maintaining project documentation

## Maintainer Guidelines

### Code Review Process

- All changes require review from at least one maintainer
- Breaking changes require review from the lead maintainer
- Security-related changes require extra scrutiny
- Reviews should be constructive and educational

### Issue Management

- Triage new issues within 48 hours
- Label issues appropriately
- Close duplicate or resolved issues
- Escalate security issues privately

### Release Process

1. Update version in package.json
2. Update CHANGELOG.md
3. Create GitHub release with release notes
4. Publish to Cloudsmith registry
5. Announce release in community channels

### Decision Making

- Minor decisions can be made by any maintainer
- Major decisions should involve discussion among all maintainers
- Breaking changes require consensus
- Use GitHub Discussions for major architectural decisions

## Becoming a Maintainer

We welcome new maintainers! To become a maintainer:

1. **Contribute Regularly**: Make consistent, quality contributions over time
2. **Show Leadership**: Help other contributors, review PRs, participate in discussions
3. **Demonstrate Expertise**: Show deep understanding of the project
4. **Community Involvement**: Active participation in project discussions and decisions

### Requirements

- History of quality contributions to the project
- Good understanding of the codebase and architecture
- Commitment to the project's goals and values
- Ability to work collaboratively with others
- Time to dedicate to maintainer responsibilities

### Nomination Process

1. Current maintainers can nominate contributors
2. Self-nomination is also welcome
3. Discussion among current maintainers
4. Consensus decision on new maintainer

## Maintainer Onboarding

New maintainers receive:

1. **Repository Access**: Write access to the main repository
2. **Documentation**: Access to maintainer-specific documentation
3. **Tool Access**: Access to CI/CD tools, package registries, etc.
4. **Communication**: Addition to maintainer communication channels
5. **Mentoring**: Pairing with experienced maintainer for first month

## Stepping Down

Maintainers can step down by:

1. Notifying other maintainers
2. Transferring any ongoing responsibilities
3. Removing access permissions
4. Being listed as emeritus maintainer (optional)

We understand that life circumstances change and respect maintainers' decisions to step back when needed.
